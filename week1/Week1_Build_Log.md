## 二、遇到的问题 & 解决方案

### 问题 1：Remix VM PUSH0 不兼容

| 项目 | 详情 |
|------|------|
| **现象** | `postMessage` 调用报 `invalid opcode` |
| **根因** | Solidity 0.8.20+ 默认编译目标包含 PUSH0 指令；Remix VM (Cancun) 不支持 |
| **尝试过的无效方法** | 换编译器版本（0.8.19 → 0.8.34），问题仍然存在 |
| **AI 建议** | 改 EVM Version → `paris` 或 `shanghai` |
| **最终解决方案** | Solidity Compiler → Advanced Config → EVM Version = `paris`，重新编译部署即可 |
| **反思** | PUSH0 是 0.8.20 引入的字节码优化，和 0.8.x 具体版本号无关；改版本号本身没用，要改编译目标 |

### 问题 2：MetaMask 不被 Remix 检测

| 项目 | 详情 |
|------|------|
| **现象** | Remix Deploy & Run 面板 Environment 下拉无 `Injected Provider` |
| **调试过程** | 控制台 `window.ethereum` 返回空 → MetaMask 未注入页面 |
| **尝试过的方案** | ① MetaMask 设置中添加 remix.ethereum.org 到已连接站点 ② 关闭其他钱包插件 ③ 关闭标签页重建 ④ 无痕模式 |

## 七、本周最重要的 3 个学习收获

### 1. PUSH0 字节码兼容性问题（技术硬知识）

> Solidity 0.8.20+ 默认编译到 Shanghai EVM（含 PUSH0），但部分模拟环境不支持。**解决方案不是改版本号，是改 EVM Version 到 paris**。这让我理解了编译器版本号和 EVM 目标是两个独立参数。

### 2. 「链上产品 vs 普通产品」的判断边界

> 以前只会说「链上更安全」。现在能清晰判断：核心区别是**资产归属（self-custody）、去信任公平性（可验证）、可组合性（permissionless composability）**。不是所有产品都该上链——先问「这个能不能用 SQL 解决？」

### 3. AI 编程的正确姿势

> AI 能快速产出代码，但**会加速犯错，也加速纠错**。PUSH0 的坑就是经典例子——AI 写了合约，我没跑就以为好了，一跑就炸。真正的效率来自「AI 出代码 → 人工验证 → 反馈给 AI 修复」的循环，不是「AI 出代码 → 直接用」。
